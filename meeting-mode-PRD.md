## 1. Why — 왜 만드는가

### 문제

Windows 기본 방해금지 모드가 있음에도 불구하고, 실제 사용자는 회의 중 알림으로 인한 사고/스트레스를 반복적으로 경험하고 있음.

### 관찰된 실제 페인포인트 (1차 사용자 인터뷰 기준)

| # | 문제 | 근본 원인 |
|---|------|-----------|
| P1 | 회의 시작 시 방해금지 켜는 걸 **까먹음** → 화면 공유에 카톡 뜸 | 수동 트리거에 의존 |
| P2 | 회의 끝나고 **다시 끄는 것도 까먹음** → MM/업무 메시지 놓침 | 자동 해제 신호 없음 |
| P3 | 방해금지 **설정 과정 자체가 귀찮음** (알림 패널 열면 내용 보임) | OS 기본 UX의 한계 |
| P4 | 앱마다 중요도가 다른데 **OS는 전부 ON/OFF만 가능** | 세밀한 제어 부재 |

### 기존 해결책이 부족한 이유

- **OS 방해금지 모드**: 수동 + 전부 끄기 / 전부 켜기만 가능
- **온라인 미팅 감지 서비스**: 줌/팀즈 프로세스 기반이라 **오프라인 미팅에선 무쓸모**
- **캘린더 기반 자동화 (맥 Focus 등)**: 갑자기 잡히는 회의 대응 불가, Windows에선 옵션 적음

## 2. Who — 타겟 사용자

### Primary (MVP 타겟)

**오프라인 회의가 잦은 사무직 직장인**
- 주간 보고, 팀 미팅 등에서 노트북으로 자료 공유
- 카카오톡/Mattermost/Slack 등 다수 메신저 병행 사용
- Windows 환경, 듀얼/트리플 모니터 사용 가능성 있음

### Secondary (확장)

- 온라인 미팅 많은 리모트 워커
- 컨설턴트, 외국계 직장인
- 프레젠테이션/데모가 잦은 세일즈·PM

## 3. What — 제품 정의

### 한 줄 요약

> **"회의 시작/종료를 자동 감지해서, OS 알림과 메신저 상태를 한 번에 제어하는 Windows 트레이 앱"**

### 제품 원칙

1. **사용자가 규칙을 정의한다** — "무엇을 회의로 볼지"는 사용자 워크플로우마다 다름. 일괄 자동화 X
2. **자동화는 조용하게, 수동은 즉각적으로** — 확실할 땐 자동, 애매할 땐 제안, 언제나 단축키로 오버라이드
3. **재진입 비용 0** — 회의 끝나면 놓친 알림을 요약해서 한 번에 보여줌
4. **가벼움** — 항상 상주하는 앱. 메모리 낮고 배터리 부담 적어야 함 (→ Tauri)

## 4. Core Features (MVP)

### 4.1. 회의 감지 트리거 (사용자 설정 가능)

초기 설정에서 체크박스로 선택, 이후 자동 동작.

| 트리거 | 감지 방식 | 자동 강도 | 구현 난이도 |
|--------|-----------|----------|-------------|
| 디스플레이 **복제(Clone) 모드** | `QueryDisplayConfig` + `WM_DISPLAYCHANGE` | 자동 ON | 🟢 쉬움 |
| **PPT 슬라이드쇼 모드** | `EnumWindows` + `screenClass` 윈도우 클래스 감지 | 자동 ON | 🟢 쉬움 |
| **전체화면 앱 실행** | 포어그라운드 윈도우 크기 = 모니터 크기 | 자동 ON | 🟡 중간 |
| **외부 디스플레이 연결** | 디스플레이 변경 이벤트 | 제안 토스트 | 🟢 쉬움 |
| **캘린더 일정 시작 (Google/Outlook)** | 캘린더 API 폴링 | 제안 토스트 | 🟡 중간 |

**자동 ON** = 감지 즉시 방해금지 활성화
**제안 토스트** = "회의 시작하셨나요? 회의 모드 켤까요?" 알림

### 4.2. 수동 트리거

- **Floating 토글 버튼** (작고 반투명, 위치 이동 가능, 숨김 옵션)
- **글로벌 단축키** (기본값 `Ctrl+Alt+M`, 커스터마이징 가능)
- **트레이 아이콘 클릭**

### 4.3. 회의 모드 ON 시 동작

- ✅ **Windows 알림 차단** (시스템 방해금지 모드 활성화)
- ✅ **Mattermost status → DND** (API 연동, Personal Access Token)
- ✅ **Slack status → DND** (`users.setPresence` + `dnd.setSnooze`)
- ⏳ **앱별 세부 제어** → TODO (6장 참고)

### 4.4. 자동 해제 트리거

다음 중 **가장 먼저 발생**하는 신호로 자동 해제:

- 디스플레이 복제 해제
- PPT 슬라이드쇼 종료
- 전체화면 앱 종료
- 캘린더 일정 종료 시간 도달
- 기본 타임아웃 (사용자 설정, 기본 60분)

### 4.5. 놓친 알림 요약

회의 모드 해제 시 토스트 알림:

```
회의 끝! 그동안 놓친 알림
• Mattermost: 3개
• Slack: 1개
• Kakao Talk: 2개
  [모두 보기]
```

클릭 시 각 앱으로 바로 이동 + 읽지 않음 상태 유지.

## 5. User Flow

### 최초 설치 플로우

```
설치 → 온보딩 (3스텝)
  1. "언제 회의라고 볼까요?" (트리거 선택)
  2. "어떤 앱 상태를 바꿀까요?" (MM/Slack 연동)
  3. "단축키 설정" (기본값 제공)
→ 트레이 상주 시작
```

### 일반 사용 플로우 (자동 감지 시)

```
[회의실 도착]
  → 노트북 HDMI 연결 → 디스플레이 복제 설정
  → [자동 감지] 회의 모드 ON
  → Windows 알림 OFF / MM DND / Slack DND
[회의 진행]
  → 알림 없음, 화면 공유 안전
[회의 종료]
  → HDMI 분리 or 디스플레이 모드 변경
  → [자동 해제] 회의 모드 OFF
  → 놓친 알림 요약 토스트
```

### 수동 사용 플로우 (갑작스런 회의)

```
[갑자기 회의 소집]
  → Ctrl+Alt+M 누르기 (1회)
  → 회의 모드 ON
[회의 종료]
  → Ctrl+Alt+M 다시 누르거나 타임아웃 대기
```

## 7. Technical Stack

| 영역 | 선택 | 이유 |
|------|------|------|
| Framework | **Tauri** | Electron 대비 가벼운 메모리(상주형 앱에 필수), Rust 네이티브 API 접근 용이 |
| Backend | Rust | Windows API 호출, 저수준 이벤트 처리 |
| Frontend | React + TypeScript | 설정 UI, 익숙한 스택 |
| Windows API | `windows` crate | 디스플레이 모드, 윈도우 enumeration, 알림 제어 |
| 외부 API | Mattermost REST, Slack Web API, Google Calendar API | OAuth/PAT 기반 |
| 배포 | GitHub Releases + 자동 업데이트 (Tauri updater) | 초기엔 무료 배포 |

### 주요 기술 리스크

- **Windows 알림 차단 API**가 제약적 → MVP는 시스템 방해금지 모드 활성화 수준으로 시작
- **앱별 차단**은 DLL 후킹이 필요할 수 있음 → v1.0에서 재검토
- **Bizbox 등 폐쇄적 그룹웨어** API 미지원 → 캘린더 CalDAV 지원 여부 확인 필요
